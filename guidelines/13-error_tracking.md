# Error Tracking Guidelines for Web Components

When developing web components, it is important to ensure that any error generated by the component is properly handled and clearly communicated to the user. The web component is responsible for displaying the error, including providing a retry button to allow users to attempt recovery.

You can find the design of the error alert in the Figma design guide.

In addition, the component must document its internal logging system as part of its quality criteria, ensuring that issues can be traced and understood.

This document defines the approach for error tracking across distributed web components while maintaining flexibility for individual teams to implement error tracking appropriate to their specific needs.

---

## Who Tracks Errors
Each web component is responsible for its own error tracking.

Web component teams are free to choose their own error tracking implementation. Use of Sentry is optional if it fits team needs.

---

## Data to Include in Error Tracking

### Essential Error Information
- **Error type/classification:** Specific exception type or error code
- **Error message:** Complete error message with relevant details
- **Stack trace:**
  - **Frontend stack traces** may be logged or sent to monitoring
  - **Backend stack traces** must **not be exposed to the client** but should be captured in secure server-side logs for debugging
- **Timestamp:** When the error occurred, with timezone information

### Contextual Information
- Method/function name: Where the error occurred
- Input parameters: Sanitized representation of inputs that caused the error
- Application state: Relevant application state at the time of the error
- Environment information: Development, staging, production, etc.
- User action: Action being performed when the error occurred
- Previous operations: Recent operations that might have contributed to the error

### System Information
- Service name and version
- Host information: OS, OS version, device brand

---

## Data to Exclude from Error Tracking

### Sensitive Information
- Personally Identifiable Information (PII): names, addresses, phone numbers
- Authentication data: passwords, tokens, session IDs
- Health information: medical data
- Unique identifiers: social security numbers, national IDs

### Security-Related Information
- Internal network details: IPs, hostnames, URLs with credentials
- Database connection strings with credentials
- Encryption or signing keys
- System paths revealing architecture

### Performance Considerations
- Large data objects or entire datasets
- Binary data
- Redundant or duplicate information

---

## Implementation Best Practices
- Implement consistent error handling across all components
- Use structured logging with appropriate log levels
- Categorize errors for easier filtering and analysis
- Sanitize sensitive data before storing or transmitting
- Implement sampling for high-volume error scenarios
- Add context incrementally as it passes through application layers
- Use correlation IDs to track request flow across multiple components
- Consider rate limiting for error reporting to prevent flooding

---

## Error Severity Levels
- **Critical:** Application/component is completely non-functional
- **Error:** Significant functionality is impaired
- **Warning:** Non-critical issues that might lead to errors
- **Info:** Important events that aren't errors but provide context

---

## Summary
- Backend stack traces must **never be exposed to the client**, but **may be logged securely on the server**.
- Frontend stack traces may be logged or sent to monitoring but **must not appear in the UI**.
- User-facing error messages should be **clear, generic, and non-technical**.
- All sensitive or security-related data must be excluded from logs and error reports.
